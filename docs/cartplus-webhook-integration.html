CartPlus webhook integration v1

How sales executives' CartPlus quotations land automatically in HVA — set-up, day-to-day operation, and troubleshooting.

Who this is for: Beakn super admins setting up + maintaining the CartPlus webhook, and developers who need to understand what the integration does behind the scenes.

What this integration does

Beakn sales executives use the CartPlus quotation portal to send quotations to customers and accept payments. Until now, execs had to re-enter every quotation manually in HVA.

With this integration, every quotation event on CartPlus triggers a webhook. HVA:

  1. Verifies the webhook came from CartPlus (HMAC-SHA256 signature).
  2. Dedupes against duplicate deliveries (envelope id).
  3. Matches the CartPlus store to an HVA city.
  4. Matches the CartPlus exec to an HVA user.
  5. Finds (or creates) the customer contact using the phone number.
  6. Creates a visit request at QUOTATION_GIVEN stage with the quotation + line items attached.
  7. Notifies the assigned exec, their captain, and admin (in-app + push + WhatsApp).

Revisions to the same order on CartPlus update the existing quotation in place (same portal order ID, no duplicates).

v1 scope: Only order.created, order.status_changed, and order.cancelled events. Payment events, fulfillment events, product changes, and coupon redemptions stay on CartPlus and are not subscribed to.

Locked behavior — what HVA does on each event

CartPlus eventHVA action
order.created Creates a new HVA visit_request at stage QUOTATION_GIVEN. Creates a contact if the phone isn't already on file. Stores quotation + line items. Fires notifications to exec, captain, and admin.
order.status_changed Looks up the existing quotation by portal order ID. Upserts line items (same portal_line_item_id = same row, updated). HVA stage does not auto-advance — the exec moves the request to ORDER_CONFIRMED manually when the customer agrees verbally.
order.cancelled Transitions the HVA request to a terminal stage with reason "Cancelled in portal" and records the raw payload on the quotation for audit.

Initial set-up (super admin)

  1. Generate a signing secret in HVA.

    Go to Admin → Settings → Integrations → CartPlus secrets. Click Generate new secret. A modal will show the secret once — copy it immediately.

    This is the only time HVA shows the full secret. After you close the modal, only an 8-character preview (first 4 + ellipsis + last 4) is visible. If you lose the secret, generate a new one (which automatically revokes the old).
  2. Configure the webhook in CartPlus.

    In the CartPlus admin: Store Settings → Webhooks → New webhook.

    Webhook Name
    Beakn HVA (or anything you like)
    Endpoint URL
    https://visits.beakn.in/api/webhooks/cartplus
    Signing Secret
    Paste the secret you copied from HVA
    Active
    true
    Events
    Subscribe to:
    • order.created
    • order.status_changed
    • order.cancelled

    Other events (payments, fulfillment, products, coupons) are NOT used by HVA yet — leave them unchecked to save log noise.

  3. Map every HVA city to a CartPlus store ID.

    Go to Admin → Settings → Integrations → CartPlus cities. For each city, look up the numeric store.id in CartPlus (top-right of each store's settings page) and paste it into the row. Hit Save.

    Unmapped cities still work — incoming orders for an unknown store.id fall back to the "Other" city and admin gets a heads-up notification. But mapping every city makes the routing precise from day one.

  4. Map every sales exec + captain to their CartPlus user ID.

    Go to Admin → Settings → Integrations → CartPlus execs. Each row is an active sales exec or captain. Look up their CartPlus user ID (the number in data.order.created_by.id of any of their orders) and paste it in. Hit Save.

    Why this matters: when a webhook arrives, HVA matches the CartPlus exec ID to the HVA user. If there's no match, the request goes to the unassigned bucket and admin gets pinged. The first ever order from a brand-new exec will fail this check — that's expected. Map them and trigger a CartPlus retry (or wait for the next event).
  5. Test it.

    In CartPlus webhook settings, use the Test button to send a sample order.created payload to HVA. HVA's payload has a "test": true flag for these — they will appear in the webhook log but won't create a real visit request.

    You should see the test event appear in Admin → Integrations → CartPlus secrets with a "Last used" timestamp updated on your active secret.

Day-to-day — what to watch

"Last webhook" column tells you who is live

On the city + exec mapping pages, every row has a Last webhook column. If a city or exec hasn't seen a webhook in 7+ days but you know they're actively selling, the mapping is probably wrong.

The unassigned bucket is the safety net

Anytime HVA can't resolve the exec (storefront orders without created_by, OR a CartPlus exec that's not yet mapped to an HVA user), the request lands in the unassigned bucket. Captains see these in their queue and can hand-assign. Admin also gets a notification so the mapping can be fixed.

Rotating the secret

Rotate the secret whenever:

  • An employee with access to admin leaves the team.
  • You suspect the secret was exposed (logged somewhere it shouldn't be, etc.).
  • Routine policy (every 6 months is reasonable for an internal integration).

In HVA's CartPlus secrets page, click Generate new secret. This automatically revokes the previously active secret. Update CartPlus's webhook config with the new secret immediately — any delay means CartPlus webhooks will fail signature verification and accumulate retries.

Troubleshooting

Webhooks not arriving in HVA

  1. Check CartPlus's webhook delivery log — are deliveries marked failed with a 4xx response?
  2. If you see 401 Unauthorized: signing secret mismatch. Re-paste the secret from HVA into CartPlus, or rotate.
  3. If you see 5xx: HVA-side error. Check CartPlus secrets page for "Last used" timestamp — if it stopped updating, the receiver might be down.

Orders land in the wrong city

Either the city wasn't mapped (orders default to "Other"), or two cities have overlapping store_id values. Check the CartPlus cities page — HVA enforces uniqueness on store_id, so a save with a duplicate ID will fail with a clear error.

Orders go to the unassigned bucket every time

The exec's HVA user isn't mapped. Check CartPlus execs and add their CartPlus user ID. Future orders auto-assign; past orders stay in the unassigned bucket and can be hand-assigned by the captain.

Quotation revisions create duplicate rows

Shouldn't happen — HVA enforces uniqueness on portal_quotation_id. If you see duplicates, check whether two webhooks have the same envelope id but different data.order.id values. Contact engineering with the webhook event IDs from the audit log.

Pay attention to "duplicate" deliveries

CartPlus retries failed deliveries (typically on 5xx). HVA dedupes by envelope id, so a retry after a transient HVA outage is harmless — the request gets created once, subsequent retries are no-ops.

Headers HVA verifies

X-CartPlus-Event
Event type, e.g. order.created
X-CartPlus-Signature
HMAC-SHA256 of the raw body, hex-encoded
X-CartPlus-Delivery
Per-delivery ID (used for dedup, logged in webhook_events.delivery_id)
Content-Type
Must be application/json

Glossary

HMAC-SHA256
Keyed hash function used to prove a request came from CartPlus and wasn't modified in transit.
Idempotency
Property that re-delivering the same webhook event produces the same outcome (no duplicates). HVA's envelope-id UNIQUE constraint enforces this.
Envelope
The top-level JSON wrapper — {id, type, store, data, created_at} — present on every CartPlus webhook.
Storefront order
An order placed by the customer directly on the CartPlus storefront, with no created_by exec. These land in the unassigned bucket in HVA.
Dead-letter queue
The set of webhook_events rows with result = 'error'. Engineering reviews these for systemic bugs.
Paise
Indian smallest currency unit: ₹1 = 100 paise. HVA stores all money as integer paise to avoid floating-point rounding.
